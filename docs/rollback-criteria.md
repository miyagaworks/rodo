## 0. ロールバックの基本概念

**ロールバック = 「変更を元に戻す」操作の総称**。本番運用で問題が起きたとき、直前の動いていた状態に戻して被害を最小化する。レベル別に以下の 4 段階がある:

| レベル | 何を戻すか | 戻す先 | 所要時間 |
|---|---|---|---|
| Level 0 | サービス利用そのもの | 紙運用・口頭運用 | 即時 |
| Level 1 | デプロイしたコード | 直前のデプロイ | 1〜3 分 |
| Level 2 | ドメインの向き先 | 旧環境 or メンテ LP | 5 分〜数時間 |
| Level 3 | データベース | バックアップ取得時点 | 30 分〜数時間 |

**重要**: shimoda は今回が初の本番運用のため、**「初日（切替当日）」と「運用フェーズ」では取りうる手段が異なる**。本ドキュメントは両者を分けて記述する。

---

## 1. このドキュメントの目的

shimoda.rodo.run の本番切替日（Phase 3 P3-01）および運用開始後に、障害発生時のロールバック実施判断を **現場で迷わず即決できる** 基準を事前に定義する。

判断基準が事前合意されていない状態で障害が起きると、初動が 30 分〜1 時間遅れ、shimoda 業務が長時間停止するリスクがある。

---

## 1A. 初日（P3-01 切替当日）と運用フェーズの違い

| 観点 | 初日 | 運用フェーズ（切替後 1 週間以降） |
|---|---|---|
| Level 1 の有効性 | **無効**（直前の動いていたデプロイが本番に存在しない） | 有効（直前の安定デプロイに即時ロールバック可） |
| Level 2 の戻す先 | **無効**（旧環境が存在しない）→ メンテ LP のみ可能 | 同上（旧環境はないが、メンテ LP は使える） |
| Level 3 の必要性 | 低い（業務データがほぼ無い） | 高い（業務データが蓄積している） |
| Level 0「紙運用復帰」の現実性 | **第一選択**（shimoda 側が紙運用慣れしている前提） | shimoda の業務スタイル次第（再合意が必要） |

→ **初日のロールバック = Level 0（紙運用復帰）が現実的な第一選択**。これを shimoda 側と事前合意することが本ドキュメントの最重要事項。

## 2. ロールバック発動基準

以下のいずれかが **30 分以上継続** した場合、ロールバック判断を発動する。

| # | 条件 | 検知方法 | 重大度 |
|---|---|---|---|
| C-01 | shimoda 管理者または隊員がログイン不可 | ユーザー通報 + Sentry エラー率急上昇 | 致命 |
| C-02 | 出動データの書き込みが失敗（500 系エラー多発） | Sentry / Vercel ログ | 致命 |
| C-03 | API の p95 レスポンス時間が 5 秒を超過 | Vercel Analytics | 重大 |
| C-04 | データ整合性異常（テナント間データ混入の疑い） | ユーザー通報 / DB 監査クエリ | 致命（即時発動、30分待たない） |
| C-05 | 写真・署名画像のアップロードが恒常的に失敗 | Sentry / ユーザー通報 | 重大 |
| C-06 | 認証セッションが頻繁に切れる（5分以内に強制ログアウト多発） | ユーザー通報 + Sentry | 重大 |

**注**: C-04（データ混入）は **即時ロールバック発動**（30 分待機なし）。データ漏洩は時間が経つほど被害拡大するため。

## 3. ロールバック判断フロー

```
障害発生
   ↓
1次切り分け（5分以内）
   - エラーログ確認（Sentry / Vercel）
   - 影響範囲確認（特定ユーザーのみか全体か）
   - 該当機能の代替手段の有無確認
   ↓
30分継続観察（C-01〜C-03, C-05, C-06 の場合）
   ↓
継続中？
 ├─ Yes → ロールバック実施判断
 └─ No  → 監視継続、ポストモーテム作成
```

### 3.1 判断責任者

- **第一判断者**: 宮川（プロジェクト責任者）
- **第二判断者**（宮川不在時）: shimoda 側責任者と協議の上、宮川の代行で判断

### 3.2 判断時の連絡

- ロールバック実施を決定したら、即座に shimoda 側責任者へ電話連絡（メール・LINE は二次手段）
- 「これからロールバックします」「業務影響時間は X 分予想」を伝達

## 4. ロールバック手順（4 段階）

ロールバックは **影響範囲が小さい順** に試す。Level 1 で復旧したら Level 2 以降は不要。

**初日と運用後で適用順序が異なる:**
- **初日（P3-01）**: Level 1 は無効 → Level 0（紙運用）を最優先 / 並行でコード修正＋再デプロイ
- **運用後**: Level 1 → Level 2 → Level 3 の順で試す

### Level 0: 紙運用への一時復帰（初日の第一選択）

| 項目 | 内容 |
|---|---|
| 操作 | shimoda 側に「本日は紙の出動記録に戻してください」と電話連絡。後日 rodo へ手入力で反映 |
| 影響範囲 | shimoda 側の業務オペレーションのみ（データ損失なし。後日反映で完全復元可） |
| 所要時間 | 連絡から数分（shimoda 側の切替判断次第） |
| 適用条件 | 初日にシステム障害発生、または運用後でも業務継続が最優先の場合 |
| 事前準備（必須） | shimoda 側に「障害時は紙運用に戻れる体制を本番切替日から最低 2 週間維持」を **契約・覚書に明記** |
| 復帰時の手順 | (1) 紙の出動記録様式を shimoda が保管 / (2) 障害復旧後、紙の内容を rodo に手入力 / (3) 入力者・入力時刻をデータに記録（監査証跡） |

### Level 1: アプリケーションレベル（運用フェーズ向け）

| 項目 | 内容 |
|---|---|
| 操作 | Vercel ダッシュボードから直前のデプロイにロールバック（Instant Rollback 機能） |
| 影響範囲 | コードのみ。DB・DNS は触らない |
| 所要時間 | 1〜3 分 |
| 適用条件 | 直近のデプロイで問題が発生した場合（コード起因） |
| **初日の扱い** | **無効**。直前の動いていたデプロイが本番に存在しないため適用不可。代わりに Level 0（紙運用）+ コード修正・再デプロイで対応 |
| 操作手順 | (1) Vercel Project → Deployments、(2) 安定していた直前デプロイの「...」メニュー、(3) Promote to Production |
| 注意（DB マイグレーションを伴うデプロイの場合） | コードを戻しても DB スキーマは新版のまま → 旧コードがスキーマに合わず動かない可能性あり。**スキーマ変更を伴う PR は緊急用ロールバック SQL を事前に用意**（P0-13 の Level 2 スキーマロールバック参照） |

### Level 2: DNS レベル（中程度）

| 項目 | 内容 |
|---|---|
| 操作 | DNS レコードを「メンテナンス中」LP に向ける（旧環境は存在しないため復元不可） |
| 影響範囲 | 全ユーザー、DNS 反映待ちが発生 |
| 所要時間 | 5 分（ただし DNS 伝播に最大数時間） |
| 適用条件 | Level 1 で復旧しない、または基盤レベルの問題、データ漏洩など即時公開停止が必要なケース |
| **初日の扱い** | 旧環境がないため「メンテ LP 表示」のみ可能。実質サービス停止 → Level 0（紙運用）と必ずセット |
| 事前準備（必須） | (1) メンテナンス中 LP の用意（静的 HTML、Vercel に別プロジェクトとしてデプロイ可）、(2) DNS の TTL を短く（300 秒）設定（P1-05 で対応） |
| 注意 | DNS 伝播は完全制御不能。即時公開停止が必要なら Vercel ダッシュボードで「ドメイン解除」のほうが早い |

### Level 3: データベースレベル（最重）

| 項目 | 内容 |
|---|---|
| 操作 | Phase 2 で確立したバックアップから point-in-time recovery |
| 影響範囲 | ロールバック時点以降のデータ消失 |
| 所要時間 | 30 分〜数時間（DB 規模次第） |
| 適用条件 | Level 1, 2 で復旧不可、かつ shimoda 業務責任者の承認 |
| **初日の扱い** | **不要**（業務データがほぼ無いため）。データ整合性異常があっても、テーブル全削除して再投入のほうが早い |
| 事前条件 | P2-05 のバックアップ・リストア手順が実機検証済み |
| 必須 | shimoda 側に「データが X 時点まで巻き戻ります」と事前同意 |

| A-05 | 同一原因の再発防止策の実装計画 | ポストモーテム内 |

## 5A. 切替当日（P3-01）の障害シナリオ別対応

| 障害状況 | 第一対応 | 並行対応 |
|---|---|---|
| デプロイ直後にログイン不可 | shimoda へ「本日は紙運用に戻してください」と電話連絡（Level 0） | コードのバグ修正 + 再デプロイ（30 分以内目標） |
| 写真アップロード不可（その他は動く） | shimoda へ「写真機能を使わずに出動記録を続けてください」と連絡 | 該当機能のみ調査・修正、翌日修正版をデプロイ |
| データ書き込み不可（DB 起因） | Level 0（紙運用復帰） | DB 接続設定・スキーマ調査、復旧後に紙の内容を手入力で反映 |
| 致命的データ漏洩疑い | **即時 Vercel ダッシュボードでドメイン解除**（サービス公開停止） + shimoda へ電話 | 影響範囲調査、原因究明、復旧計画策定 |
| 一部ユーザーのみログイン不可 | 当該ユーザーへ「他のユーザーで作業してください」と連絡 | DB のユーザーレコード調査、該当ユーザー復旧 |

## 7. 想定される判断ミスと予防

| 判断ミス | 予防策 |
|---|---|
| 「もう少し様子見」で 1 時間放置 | C-01〜C-03 は **30 分継続でロールバック** を機械的に適用（迷わない） |
| Level 3 を最初から実施してデータ消失 | Level 1 → 2 → 3 の順序を厳守。Level 3 は shimoda 承認必須 |
| 復旧後にポストモーテム作成を後回し | 24 時間以内ルールで強制 |
| shimoda への連絡が遅れて不信感 | 電話連絡を一次手段、メール・LINE は二次手段 |

## 7B. shimoda との事前合意項目（最重要）

本ロールバック戦略は **shimoda 側の協力なしに成立しない**。以下を契約・覚書で事前合意する。

| # | 合意項目 | 必要時期 |
|---|---|---|
| AG-01 | 障害時に紙運用に戻れる体制を、本番切替日から最低 **2 週間** 維持する | 切替日 1 週間前まで |
| AG-02 | 障害発生時、宮川からの「紙運用復帰」連絡があれば即時切替に応じる | 切替日 1 週間前まで |
| AG-03 | 紙運用に戻った場合、復旧後の手入力作業に協力する（入力者・入力時刻を記録する） | 切替日 1 週間前まで |
| AG-04 | 致命的データ漏洩疑いの際、サービス即時停止の判断を宮川に委ねる | 切替日 1 週間前まで |
| AG-05 | 24 時間以内のポストモーテム作成期間中、shimoda 側の業務継続方法を協議する | 切替日 1 週間前まで |
| AG-06 | 切替日（P3-01）の日付・時間帯・宮川の張り付き予定 | 切替日 2 週間前まで |
| AG-07 | 本番切替日から最初の 1 週間は、宮川が朝夕にステータス確認連絡を入れる | 切替日 1 週間前まで |

→ これらは `docs/incident-response.md` および `docs/pre-launch-todo.md` の C 群（法務・契約系）と連動する。覚書として明文化すること。
