## 1. 計画の目的とスコープ

### 1.1 達成したい状態

- `shimoda.rodo.run` で株式会社shimodaの本番運用が開始できる状態
- 同じコードベースで `<新顧客>.rodo.run` を後から追加できるサブドメインベースの基盤が整っている
- 「フィーチャーフラグの粒度」「テナント設定項目」を後から拡張できる土台ができている
- 買取型（コードフォーク譲渡）にもそのまま転用できる単一コードベース構造を維持している
- 本番障害時にロールバック可能な手順が文書化されている

### 1.2 スコープ内（この計画書の対象）

- Phase 0〜5 の技術タスク設計
- 本番インフラ（Vercel + Cloudflare + DB）の選定と切替手順
- マルチテナント基盤（サブドメイン判定、認証、データ分離）のリファクタ計画
- 本番移行のロールバック計画
- 運用観察ルーチンの定義

### 1.3 スコープ外（別途計画化が必要）

- フィーチャーフラグの具体仕様（下田社長ヒアリング後に確定）
- 料金プラン・課金基盤の実装（ヒアリング後）
- テナント作成管理画面のUI詳細設計
- DISPLAY_CONFIG・地名/高速マスタの並び順最適化（運用観察後）
- 買取型顧客向けのフォーク譲渡フロー設計
- マーケティング・販売活動

### 2.4 技術調査で確認した重要事実（2026-04-20 調査）

- **Vercel ワイルドカードドメイン**: `*.rodo.run` の登録自体はHobby/Pro 共通で可能だが、**Vercelネームサーバー方式が必須**。Hobby は1プロジェクト50ドメインまで（`*.rodo.run` 1件で十分カバー可）。Pro は Team metering（メンバー単位月額$20）の課金体系。本案件単独ならHobby上限で足りる可能性あり。要最終確認
- **Cloudflareの位置づけ**: Vercel公式は「Cloudflareを前段に置くこと」を推奨しない。DNS-onlyでもVercel Firewall が動作しなくなる。さらに **Cloudflare はワイルドカードCNAMEをProxy できない（Enterpriseのみ）**ため、`*.rodo.run` を Cloudflareでホストする場合、結局 DNS-only にする以外選択肢がない
- **NextAuth v5 の AUTH_TRUST_HOST**: `AUTH_URL` を設定せず `AUTH_TRUST_HOST=true` でリクエストヘッダから動的に host を取得する方式が公式推奨。ただし v4 と v5 で挙動差があり、サブドメイン環境で issue が継続報告されている。lazy init パターン（NextAuth に関数を渡し、リクエストごとに configを返す）の採用を要検討

## 8. ロールバック計画

### 8.1 ロールバック判断基準（J-12 と整合）

以下のいずれかが**30分以上継続**した場合、自動的にロールバック判断を発動:
- shimoda 管理者または隊員がログイン不可
- 出動データの書き込みが失敗
- p95 レスポンス時間が 5秒を超過
- データ整合性異常（テナント間データ混入の疑い）

### 8.2 ロールバック手順

**Level 1: アプリケーションレベル（最も軽量）**
- Vercel ダッシュボードから直前のデプロイにロールバック（Instant Rollback機能）
- 影響範囲: コードのみ。DB・DNSは触らない
- 所要時間: 1〜3分

**Level 2: DNSレベル（中程度）**
- DNSレコードを旧環境（または「メンテナンス中」LP）に戻す
- 影響範囲: 全ユーザー、DNS反映待ち発生
- 所要時間: 5分（ただしDNS伝播に最大数時間）

**Level 3: データベースレベル（最重）**
- Phase 2 で確立したバックアップから point-in-time recovery
- 影響範囲: ロールバック時点以降のデータ消失
- 所要時間: 30分〜数時間（DB規模次第）
- 事前条件: Level 1, 2 で復旧不可と判断、かつshimoda業務責任者の承認

### 8.3 ロールバック実施後の必須アクション

1. shimoda側責任者への即時連絡（電話）
2. 障害ポストモーテム（24時間以内）
3. 再切替日程の再策定（最短でも1週間後）
4. 同一原因の再発防止策の実装
